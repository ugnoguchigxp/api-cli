# api-cli MCP Interaction Skill

このスキルは、`api-cli` を MCP サーバーとして使用して、登録された各種プロバイダーの API を呼び出すためのガイドラインを AI に提供します。

## 概要

`api-cli` は、ローカルで管理されている認証情報（OAuth2 PKCE, API Key）を使用して外部 API を呼び出す MCP サーバーを提供します。AI は、ユーザーの代わりに（またはユーザーの承認を得て）これらの API を実行できます。

## 利用可能なツール

### 1. `list_providers`
登録されているプロバイダー（サービス名）の一覧を取得します。API呼び出しの前に、対象のプロバイダーが正しく登録されているかを確認するために使用します。

### 2. `api_call`
連携先の API に対して HTTP リクエストを実行します。
- `provider_id`: 対象プロバイダーの識別子（`list_providers` で取得可能）。
- `method`: HTTP メソッド (GET, POST, PUT, DELETE, etc.)。
- `path`: ベースURL以降のパス。
- `body`: (JSON形式) リクエストボディ。

## 重要なガイダンス

### 1. セキュリティと明示的な承認
- **手動承認のエラー**: `api_call` 実行時に「Action required: ... manual approval」というエラーが返された場合、そのエンドポイントへのアクセスはユーザーによる手動の承認が必要です。 
  - **アクション**: ユーザーに表示された `api-cli mcp approve ...` コマンドを実行して承認するように促してください。
- **機密情報の保護**: API レスポンスにトークンや個人情報、接続シークレットが含まれる可能性があります。これらを不必要にログに出力したり、ユーザーへの回答に含めたりしないでください。

### 2. 認証エラーのハンドリング
- **AuthRequired / AuthExpired**: 認証が必要または期限切れの場合、AI は直接解決できません。
  - **アクション**: ユーザーに `api-cli auth login <provider_id>` を実行して認証を更新するように依頼してください。

### 3. エラー時の自己修復
- エンドポイントのパスが間違っている可能性がある場合は、まず `list_providers` や基本となるパス（`/` など）を確認して、推測ではなく確実な情報に基づいてリクエストを再構築してください。

## 利用例

### シナリオ: 特定のサービスのユーザー情報を取得する
1. `list_providers` を呼び出し、対象サービス（例: `github`）がリストにあるか確認。
2. `api_call` で `provider_id="github"`, `method="GET"`, `path="/user"` を実行。
3. エラーがなければ取得したデータを要約してユーザーに提示。
